Harness 架构重构 — 三个方向

从"单向接入"到"可插拔平台"

当前状态

Agent Loop → harness.execute_tool() → HookEngine.pre → PermissionGate → ToolExecutor → HookEngine.post
问题：Harness 只参与了工具执行，skills 扫描了但 prompt 不走它，MCP 完全没接，permission 规则不持久化
A

渐进增强

保持现有 Harness 接口不变，逐个模块增强。SkillLoader 接真实 registry，HookEngine 加可视化编辑器，PermissionGate 加规则持久化。改动小，风险低，但 Harness 的 API 设计可能不够干净。

B

事件总线重构

用 EventBus 作为中枢，所有子系统（Agent、Skill、Hook、MCP、Permission）都通过事件通信。Harness 本身变薄，只负责初始化各子系统。优点是解耦彻底，缺点是大量重写。

C

Capability Registry

把 Skills、Hooks、MCP、Tools 统一为"Capability"抽象，用一个 Registry 管理。每种 Capability 有自己的生命周期（install/enable/disable/configure/upgrade）。Harness 变成 Capability 的调度器。最干净的抽象，但设计成本最高。

推荐 B + C 的混合：用 EventBus 解耦通信，用 Capability trait 统一 Skills/Hooks/MCP/Tools 的管理。这是 Claude Code 和 Hermes 融合后的自然演进方向。
